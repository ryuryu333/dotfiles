# Nickel の設計方針

- イミュータブルな値の変換と関数合成を中心に設計する。
- 設定値はレコード形式で管理し、レコード合成で統合する。
- ドメイン知識や状態はスキーマで表現・制約する。

## 命名規則

- 変数・関数・引数・レコードフィールドは `lower_snake_case`。
- 型・コントラクト・enum タグは `UpperCamelCase`。
- 外部仕様で名前が定められたレコードフィールドは、その仕様の命名を維持する。

## import 規則

- import はファイル先頭の `let` にまとめる。
- import の形式は、使用箇所から参照元を判別できることを基準に選ぶ。
- コード量が多く、分割束縛では import 由来か判別しにくい場合は、モジュール全体を束縛してフィールド参照する。
- 短いスコープで参照元が明瞭な場合は、分割束縛を使用してよい。

## 型注釈・契約規約

- 関数や再利用可能な計算のインターフェースには、型注釈 `:` を使う。
- 設定データ、レコード、配列、スキーマの検証には、契約 `|` を使う。
- 値域、形式、正規表現など、静的型で表現できない条件にはカスタム契約を使う。
