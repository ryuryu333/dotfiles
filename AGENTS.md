# AGENTS.md

## ユーザーとの対話
### 原則

- agent は user の承認無しでは実装作業は実施しない。
- 設計・方針の承認を得ない状態では、実装作業を開始しない。

### 基本形式

- user: 実装したい機能の概要提示、相談事項の提示。
- agent: 要件定義、方針検討、設計を実施。
- agent: user に回答。
- user: フィードバック。
- agent: 必要に応じて、要件定義、方針検討、設計をやり直す。
- agent: user に回答。
- user: 承認。
- agent: 実装、検証を実施。
- agent: user に回答。

### 回答フォーマット

- 以下の流れで回答する。（）表記は不要な場合は省略可能。
  - （相談） > 結論 > 理由
  - NG: 結論 xxxAPI を内部で呼びだすラッパーとして実装します。理由 ...。以上から、実装では API に渡す hoge は 1 と仮定します。よろしいですか。
  Ok: 相談 実装に用いてる xxxAPI では hoge の指定が必須です。1 と仮定してもよろしいですか。結論 xxAPI を内部で呼びだすラッパーとして実装します。理由 ...。
- user が差分画面で確認できる情報は延べない。
  - NG: GetWWeather -> GetTodayWWeather に変更。
  - OK: 天気取得関数 GetWWeather を実態に合わせた名前に変更。当日の天気取得と明示するために Today を関数名に追加。
  - 理由: GetTodayWWeather という関数名は変更差分画面で確認できる。変更の意図は差分画面では分からない。
- 具体的な検証データは述べず、概要を伝える。
  - example 1
    - NG: task test 実行、pass 10, fail 0 でした。
    - OK: テストは全て通過しました。
    - 理由: User が知りたいのはテストが成功か失敗かという事実。
  - example 2
    - NG: gcloud projects list の結果、name:xxxpj ID:xxx ...（具体的な結果を羅列）。そのため、xxxpj の someSetting は true です。 
    - OK: gcloud 経由で GCP のプロジェクト一覧を調査しました。xxxpj の someSetting が true になっていました。
    - 理由: OK 例の回答をした後、User が詳細情報を求めた場合、コマンドと実行結果の生データを伝える。一度の会話で全てを伝えるのではなく、対話を通じて情報を伝えていく。
- ファイルリンク。URL リンク、コードスペースは利用しない。
  - 詳細を要求された場合は許可。
  - example 1
    - NG: config/app/git/default.nix 内の user は...。
    - OK: config の app で定義されている git の user は...。
  - example 2
    - NG: ...です。公式ガイド<具体的な URL>
    - OK: Microsoft Learn の F# のコーディング規則によると...です。
  - example 3
    - NG: 以下のインターフェースを定義します。<コードスペース>。
    - OK: Hoge クラスを追加します。責務は...です。状態を保持するために、enum で認証前・途中・後を表現します。
- 次の機能は使わない。
  - 表
  - WEB リンク埋め込み
  - ファイルリンク埋め込み


## 基本原則

- YAGNI、KISS、DRY、単一責任原則。
- 変数や関数の命名は、簡略さよりも明示性を優先する。
  - NG: id, GetWetm
  - OK: user_id, GetTodayWWeather
- エラー、不正操作は仕組み・設計で防ぐ。
  - example 1
    - NG: 処理の途中でエラーを catch して代替値を渡す。
    - OK: 不正な処理は原則エラーにする。
  - example 2
    NG: 関数内の処理途中で null チェック。
    OK: 関数境界で検出・制限する。e.g. 型で制約。データを検証機構経由で受け渡しする。
- 後方互換性よりも全体設計の保守性を優先する。
  - e.g. calss hoge の 公開関数 fuga の名称を piyo に変更する
  - NG: public piyo, public fuga
  - OK: public piyo のみ。fuga を呼び出し箇所を piyo に変更。

## 各工程の規則
### 要件定義

- 必要な情報を都度収集する。
- 設計を始める前に、User の要件と期待する結果を明確にする。
- 要件が不明瞭な場合。
  - 設計に影響を与えない内容なら推測で補完する。user への報告には何を推測したかを含める。
    - e.g. 天気取得機能実装にあたり、対象地域が不明瞭だったので、東京として仮定しました。
  - 方針、設計に影響する場合、残りの要件定義作業が終わった後、作業を中断して user に相談する。

### 方針検討

- 要件が複雑・複合的であった場合、レビューや検証が容易な小さい単位に分割する。
  - example
    - user input: アプリに天気取得機能を追加して。
    - 要件を 3 分割する。天気取得のロジック実装（機能単体の関数、単体テスト）、アプリ組み込みの準備（インターフェース等）、アプリ組み込み（アプリ側の設計変更、統合テスト）。
- 要件を分割した場合、User に伝える。
  - example
    - user input: アプリに天気取得機能を追加して。
    - agnet: アプリ組み込みの前段階として、天気取得関数の実装から行います。 

### 設計

- 設計では以下を決定する
  - 追加・変更・削除するモジュール、クラス、関数、データ型、設定項目
  - それぞれの依存関係
  - 公開インターフェース
  - エラーの検出・通知方法
  - 編集範囲
  - 破壊的変更の有無
  - 検証方針
- 複数の実装案を作成する。
- 以下の観点で各案を評価する。
  - 基本原則との適合性。
  - ユーザー要件との適合性。
  - 既存実装との統一性
- 採用案を決定する。

### 実装
- 設計で決めた範囲以外は編集しない。

### 検証
- fmt や test を実施する。
  - エラーが発生した場合、原因を調査する。
    - 設計を変更しない、かつ、編集範囲内の場合（e.g. Typo）、修正は許可無しで実施可能。
    - 上記を満たさない場合、user の承認外となるため修正を中断する。user に相談する（e.g. 利用する外部インターフェースの実装ミス）。
- 事前に決定した設計に適合しているか評価する。
  - 不適合の場合、設計を変更せず編集範囲内で修正できるなら修正する。それ以外は user に相談する

## コマンド実行

- nix develop .#default を除き、全てのコマンドは以下のフローで実行可否を判断する。

### 実行可否の判断
#### step 1

- ファイルを編集するコマンドの場合、設計で承認された編集範囲内か確認する。
  - 編集範囲外の場合は実行せず、必要なら user に相談する。
- 編集範囲内またはファイルを編集しない場合、step 2 に進む。

#### step 2

- 禁止するコマンドの性質に該当するか確認する。
- 明示された例外コマンドであるか確認する。
- 禁止かつ例外適用外の場合、user の許可を求める。
- 禁止されていないコマンド、または、例外コマンドなら step 3 に進む。

#### step 3
- nix develop .#default で開発環境に入る。
- devshell にてコマンドを実行する。
- コマンドが devshell に存在しない場合、user に相談する。

### user によるコマンド許可

- user の許可を得たコマンドは例外許可されたコマンドと同等に扱う。
- user の許可は、承認を受けた次の agent ターン中、承認された目的・対象の範囲に限り有効。
- 以降のターンで再実行する場合は、再度許可を得る。

### 禁止するコマンドの性質
- 開発環境やシステム構成を変更する。
- 依存関係を追加、削除、更新する。
- バージョン管理の履歴やリモート状態を変更する。
- 外部サービスの状態を変更する。
- ファイルやデータを削除する
- 管理者権限へ昇格する。
- 影響範囲や変更内容を実行前に特定できない。


### 例外許可コマンド

- 編集範囲の fmt
- 外部状態を変えない test
- 個別ディレクトの AGENTS.md にて定義された例外許可コマンド
